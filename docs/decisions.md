
# 聖地巡礼マップ 意思決定ログ

## 背景

ラブライブ！学会のシステムとして、シリーズ横断の聖地巡礼情報マップを構築する。
ただ聖地マップを作るだけでなく、万が一ビューワーが潰れた後もデータが生き続けること、他の開発者がビューワーを作れることも目標とする。

---

## 意思決定

### データフォーマット: GeoJSON

**決定**: GeoJSON (RFC 7946) を採用する。

**理由**:
- 地理情報のオープンスタンダード。
- GitHubがGeoJSONをリポジトリ上でそのままマップ表示する。
- 誰でもAPIやファイルとして取得・再利用できる。

---

### ファイル構成: シリーズごとに1ファイル

**決定**: `nijigasaki.geojson`, `sunshine.geojson` のようにシリーズ単位でファイルを分ける。

**理由**:
- 1ファイルにまとめると、複数PRが同時にマージされるときコンフリクトが必ず発生する。
- ファイルを1エントリ1ファイルにすると、PR差分が煩雑になりレビューしにくい。
- シリーズ数は固定に近いのでファイル数が無限に増えない。

---

### 投稿インターフェース: Discord スラッシュコマンド

**決定**: Discordのスラッシュコマンド(`/spot`)で投稿を受け付ける。

**理由**:
- 学会のコミュニティはDiscordに存在する。
- 既存フォームとは別に、コミュニティメンバーが日常的に使うツールで完結させる。
- スラッシュコマンドはATTACHMENTオプション(type 11)で画像も受け取れる。

**却下した選択肢**:
- Webフォーム: 新たな導線が必要、DiscordのUXと分離する。
- GitHub Issues/PR直接: 一般ユーザーにGitHubアカウントを求めるのは現実的でない。

---

### アウトプット: GitHub PR (直接commitしない)

**決定**: サーバーはGitHubに直接commitせず、PRを作成する。

**理由**:
- 人間のレビューを挟むことで、誤った座標・不適切な画像・重複スポットを弾ける。
- モデレーションコストを最小化しながら品質を担保できる唯一の構造。
- 有名スポットへの重複投稿も、PRレビュー時に近隣ポイントの警告を出せば対処できる。

---

### スパム対策: サーバー参加3日フィルタ

**決定**: Discordサーバーへの参加から3日未満のユーザーの投稿を拒否する。

**理由**:
- アカウントレスのモデレーション手段として最もコストが低い。
- 荒らし目的の新規アカウントを弾ける。
- 3日はコミュニティへの最低限のコミットメントを示す閾値として妥当。

---

### 画像の保存先: GitHubリポジトリ

**決定**: 画像はDeno Deploy上で圧縮してGitHubリポジトリに保存する。

**理由**:
- データとビューワーを分離するにあたり、画像もデータと同じ場所に持つべき。
- 外部ストレージサービスに依存すると、そのサービスが死んだときに画像が消える。
- GitHubに画像を置けば、リポジトリをforkした人が画像ごと持ち出せる。

**前提条件**: 縦720px以下・JPEG・品質80%に圧縮し、サイズを抑える。

---

### GeoJSONのhash: 持たない

**決定**: hashはGeoJSONに含めない。インデクサーDB側で算出・管理する。

**理由**:
- hashはDBの派生情報であり、データではない。
- GeoJSONに含めると、直接編集する貢献者がhashを維持する責任が生まれ運用が壊れる。

---

### シリーズの管理: enum (series.json)

**決定**: seriesフィールドはリポジトリの`series.json`で管理するenumとする。

**理由**:
- 自由入力にすると表記ゆれが発生し、シリーズでのフィルタリングが機能しない。
- 新シリーズの追加はファイル1枚の更新で済む。

---

### エピソードの管理: 自由文字列

**決定**: episodeフィールドは自由文字列とする (例: "1期3話", "劇場版", "スクスタ")。

**理由**:
- シーズン・話数・映画・ゲームなど形式が多様で構造化が困難。
- episodeは主に表示用途であり、検索の主軸はseriesで十分。
- 構造化することで投稿ハードルが上がるデメリットが上回る。

---

### GitHub連携: GitHub Apps

**決定**: GitHubとの連携にはGitHub Apps認証を使用する。

**理由**:
- Personal Access Tokenは特定の個人アカウントに紐づくため、その人が学会を離れるとトークンが失効する。
- GitHub Appsはリポジトリにインストールする形で認証するため、個人に依存しない。
- Installation Tokenは短命(1時間)で、権限スコープも最小限に絞れる。

---

### 画像処理: JSR @matmen/imagescript

**決定**: 画像処理にJSRの `@matmen/imagescript` を使用する。

**理由**:
- Deno Deployで動作する。
- `resize` と `encodeJPEG` が揃っている。
- JPEG再エンコードによりEXIFが除去される。

---

### インデクサーDB: プロトタイプスコープ外

**決定**: 今回のプロトタイプではインデクサーDBは実装しない。

**理由**:
- 投稿フロー(Discord → Deno Deploy → GitHub PR)の検証を優先する。
- インデクサーとビューワーは後続フェーズで設計する。

---

## 今後の検討事項 (スコープ外)

- インデクサーDBの選定 (Turso など)
- 差分同期の実装 (日次バッチ + コミットSHAトラッキング)
- 重複スポットの検出と警告 (PR作成時に近隣ポイントをコメント)
- ビューワーの設計
- 既存マップ運用者からのバルクインポート対応
